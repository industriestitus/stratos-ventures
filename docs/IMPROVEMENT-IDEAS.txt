1. Monolitikus index.html felbontása
   Prioritás: Fontos | Effort: ~4+ session

   14,634 sor és 544 függvény egyetlen fájlban — ez a legnagyobb technikai
   adósság. ES modules-re bontás (pl. portfolio.js, dashboard.js, company.js,
   utils.js) drasztikusan javítaná a karbantarthatóságot, a debug-olást és a
   betöltési időt.

   Miért fontos: Egy bug keresése 14K sorban exponenciálisan nehezebb, mint
   moduláris fájlokban. A böngésző sem tud párhuzamosan cache-elni.

2. Tesztek hozzáadása
   Prioritás: Közepes | Effort: ~4+ session

   Nincs egyetlen automatizált teszt sem. A 354+ bug fix mind manuális QA-ból
   jött. Legalább a core üzleti logikára (P&L, TWR/XIRR, DCF, scoring) kellene
   unit test, hogy refaktoráláskor ne törjön semmi.

   Miért fontos: Jelenleg minden módosítás után manuális böngésző-tesztelés
   kell — ez nem skálázható.

3. Build tooling (Vite/esbuild) — MAYBE LATER
   Prioritás: Alacsony | Effort: ~2-3 session
   Státusz: Elhalasztva — csak a #1 (monolith felbontás) részeként releváns.
   GitHub Pages gzip + SW cache-first jelenleg elegendő.

   Ha a fájlokat szétbontjuk, egy build tool kezelné a bundling-ot,
   minification-t, és cache-busting-ot. Vite dev server-rel HMR-t is kapnánk.

   Miért fontos: Jelenleg a Service Worker verzió-bumpelése az egyetlen cache
   invalidáció.

8. Cloud sync conflict resolution — MAYBE LATER
   Prioritás: Fontos | Effort: ~4+ session
   Státusz: Elhalasztva — single-user használat mellett a last-write-wins
   kockázata alacsony. A 1.5s debounce ablak szűk, a multi-device egyidejű
   szerkesztés ritka. Ha multi-device használat nő, érdemes _version counter
   + 409 conflict response megoldást bevezetni.

   Jelenleg last-write-wins — ha két tab-ról vagy eszközről módosítasz, az
   utolsó mentés nyer. Kellene egy merge stratégia (pl. field-level timestamp,
   3-way merge, vagy legalább conflict detection + user prompt).

   Miért fontos: Ahogy az app-ot több eszközről használod (PWA desktop +
   mobil), az adatvesztés kockázata nő.

9. Multi-tab localStorage sync — MAYBE LATER
   Prioritás: Közepes | Effort: ~1 session
   Státusz: Elhalasztva — D1 módban a StorageEvent listener szándékosan
   kikapcsolt (szerver az igazság forrása). Az egyetlen kliens-only adat
   (price alerts) ritkán változik. Nincs valódi adatvesztés, csak átmeneti
   megjelenítési késés.

   StorageEvent listener hozzáadása, hogy ha két tab nyitva van, ne olvassanak
   stale adatot. A KNOWN-ISSUES-ben már szerepel.

   Miért fontos: Két tab-ból való szerkesztésnél az egyik felülírhatja a másik
   változtatásait.

10. Staging/test D1 környezet
    Prioritás: Közepes | Effort: ~1 session

    Egy második D1 database a fejlesztéshez, hogy a schema módosítások ne az

    Miért fontos: Jelenleg minden D1 módosítás közvetlenül az éles
    adatbázisodon fut — ez kockázatos.
